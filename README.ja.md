## BusyBeaverとは？

BusyBeaverは、LLMによって構築されたナレッジベースであり、そのミッションは、人類にとって最も重要な知識——主に書籍から得られる古典——を取り込み、**エージェントにやさしい**ナレッジベースへと変換することです。

私たちはこれを、知識を小さく、配布可能で、豊かな意味を持つ断片へと分解することで実現します。私たちはそれぞれの断片を**Meme**と呼びます。

> **Meme**という言葉は、イギリスの生物学者**Richard Dawkins**が*The Selfish Gene*（利己的な遺伝子）（1976年）の中で作り出したもので、彼はそれを**文化伝達の基本単位**と定義しました。BusyBeaverはこのアイデアを文字通り実践します。Memeとは、移動し、複製し、再結合できるほど小さな知識の単位なのです。

- すべての**Meme**は独自の**メタデータ**を持ちます。
- すべての**Meme**は**クエリ可能**です。
- **Meme**は**自由に組み合わせて**、wiki、ブリーフィング、あるいは回答へと構成できます。

このリポジトリは**ダンプ**です。数百冊の古典書籍をmemify（Meme化）して分解したものです。このコーパスは**急速に成長しており**、最初のマイルストーンは**10,000冊**です。

> **なぜ「Busy Beaver」なのか？** 計算可能性理論において、**Busy Beaver function**は*最も急速に増大する*関数の典型例であり、計算可能などんな関数をも凌駕します。この名前の背後にある野心はまさにそれです。来るべきエージェントの世界のための**超ナレッジベース**を、可能な限り速く成長させること。

エージェントをBusyBeaverに向ければ、古典を*劇的に速く*消化できます。なぜなら、テキストを分解し、分析し、リンクづけるという困難な作業は、すでに完了しているからです。

## なぜ存在するのか

私たちは、**Semantic Web**と**Knowledge Graphs**が果たせなかった約束を引き継ぎつつ、従来の知識表現スタックの限界を脱ぎ捨てたいと考えています。

決定的な転換点は、**知識の消費者が今やエージェントである**ということです。

そのため、知識はもはやRDFトリプルや形式論理表現としてエンコードする必要はありません。代わりに、エージェントが実際に最も得意とする消費形態——**豊かな意味を持つ自然言語の文とメタデータを組み合わせたもの**——を、エージェントが自由に分割し、再結合し、推論できる断片へと切り分けることができます。

> 古典的な知識表現は、人間の意味を狭い形式文法の中に押し込めていました。
> BusyBeaverは意味を言語の中に留め、構造をメタデータの中に生かします。

## 系譜と先行研究

BusyBeaverは、それ以前に存在した偉大な構造化知識プロジェクトの肩の上に立っています。私たちは特に以下に負うところが大きいです。

- **[Wikidata](https://www.wikidata.org/)** — 大規模な、協働的・多言語・機械可読な知識。
- **[DBpedia](https://www.dbpedia.org/)** — 世界の百科事典から構造化知識を抽出する。
- **[Freebase](https://en.wikipedia.org/wiki/Freebase_(database))** — エンティティと関係のオープングラフ。
- **[Semantic MediaWiki](https://www.semantic-mediawiki.org/)** — wiki自体にセマンティックな構造をもたらす。

BusyBeaverを構築するために、私たちは独自の**新しいセマンティックwiki手法**を設計しました。それは、知識の単位が硬直したトリプルではなく、出典に紐づいた、エージェントが消費可能なMemeであるという手法です。この手法は**別途オープンソース化される予定です**。

## 難題：ハルシネーションと来歴

LLMを用いてナレッジベースを構築する際の最も難しい部分は、**ハルシネーションの検出と制御**です。

BusyBeaverの答えは**来歴（プロブナンス）管理**です。私たちは知識の*出所*をできる限り忠実に保持し、**すべての主張がそれがどこから来たのかまで遡って追跡できる**ようにします——通常は特定の書籍の特定の一節です。

各知識ソースには**PN——Purple Number**が割り当てられます。

> Purple Numberは**Doug Engelbartの理想**です。すなわち、*世界のあらゆる知識の断片にはアドレスがあるべきだ*という理想です。

PNがあることで、BusyBeaverにおける主張は決して宙に浮いた記述ではありません。それはアンカーされており——アドレス指定可能で、検証可能で、出典まで引用可能です。

## Memeの構造

Memeは、BusyBeaverの原子単位です。概念的には、それぞれのMemeは次を束ねています。

| 部分 | 説明 |
| --- | --- |
| **Assertion（主張）** | 自己完結的で、豊かな意味を持つ自然言語による知識の記述。 |
| **Metadata（メタデータ）** | 検索や構成のための、種別・トピック・エンティティ・関係、その他の構造化された情報。 |
| **Provenance（来歴、PN）** | 主張をその出典の一節に結びつけるPurple Numberで、その主張を追跡可能な状態に保つ。 |

すべてのMemeが小さく、自己記述的で、出典に紐づいているため、Memeは**配布可能**（本全体を送らずに一つだけを共有できる）であり、**組み合わせ可能**（多数を組み合わせてwikiページ、ブリーフィング、あるいは回答を組み立てられる）です。

## 📦 知識のパケット交換

グローバルなインフラの飛躍は、いずれも新しい**交換の単位**から生まれました：

- **データのパケット交換**は情報を小さくルーティング可能なパケットに分割し——**インターネット**をもたらしました。
- **コンテナ輸送**は貨物を標準化され交換可能な箱に分割し——**貿易**をグローバル化しました。

**Meme は知識にとってのその単位**です。小さく、自己記述的で、出典に紐づいたパケットであり、システムや組織を越えてルーティング・交換・集約・再結合できます。**知識のパケット交換**を実現すること——それが私たちが**グローバルな知識インフラ**を築く方法です。

## なぜ生のテキストではなくナレッジベースなのか？

エージェントが読めるなら、本をそのまま与えればいいのでは？ それは違います。BusyBeaverは、生のテキストにはできない二つのことを提供します。

### 1. ハルシネーションを制御する来歴フレームワーク
生のテキストをそのままLLMに再び通すと、LLMベースのナレッジベースの最大の欠陥である**ハルシネーション**への扉が再び開いてしまいます。BusyBeaverはすべての主張を**来歴トラッキング**で包みます——各主張はそのPNによって、それが由来する出典の一節に紐づけられています。これにより、あなたはエージェントの発言を信じるしかないのではなく、それを*検証する*ためのフレームワークを手に入れます。

### 2. 出来合いのセマンティックな骨格
私たちはすでに、知識を**分解**し、それを**元のテキストにリンクバック**するという体系的な作業を行いました。散文の中に埋もれていた鍵となる意味構造は、**concept（概念）とproperty（属性）**として抽出されており、すぐに構築の土台として使えるセマンティックな骨格になっています。これは、生のテキストから出発してすべてを自分で再構築するよりも、はるかに少ない作業で済みます。

> 生のテキストは、あなたのエージェントに*読むための言葉*を与えます。
> BusyBeaverは、*すでに構造化され相互リンクされた、信頼できる知識*を与えます。

### 🌿 草 vs. 🥩 肉

こう考えてみてください。**生のテキストは草であり、BusyBeaverのKBは肉です。**

草を直接食べることは*できます*——しかしそれには、それを分解するための大きく高価な消化システムが必要です。代わりに肉を食べれば、草を消化するために費やしていたはずのエネルギーを節約できます。そのエネルギーとは、あなたのエージェントにとって**LLMトークン**のことであり、それをもっと賢いことのために使えるのです。

**肉とは高密度な知識のことです。コンテキストが高度に圧縮されています。** BusyBeaverはすでに消化を終えているので、あなたのエージェントがそれをする必要はありません。

## 使い方

BusyBeaverは**特定のストレージや検索方式を強制しません**。すべてのテキストとすべての派生知識は、すでに慎重にセグメント化され、分解され、分析されています——だからあなたが既に運用しているどんなスタックにも組み込むことができます。

### 🔎 ベクトルストア / RAG
Memeをインデックス化し、意味的類似度で検索する。断片はすでに埋め込みに適したサイズにクリーンアップ済みなので、追加のチャンク分割パイプラインは不要です。

### 🗂️ クラシックな検索（Elasticsearchなど）
MemeとそのメタデータをElasticsearch、あるいはその他の従来型検索エンジンにロードし、フィールド、ファセット、全文検索でクエリする。

### 🤖 コーディングエージェント（例：Claude Code）
**Claude Code**のようなコーディングエージェントをこのリポジトリに直接向けさせ、KBを直接読ませて、コーパス全体を横断して分析・相互参照・結論の導出をさせる。

どの経路を選んでも、PNの来歴は知識とともに移動するため、あなたのエージェントの結論は**監査可能**なままです。

## 🛠️ BusyBeaver で何ができるか

- **知識の二次創作を行う。** 知識を素材として、ブログ、PPT、小説、脚本、論文、漫画、動画などを創作できます。
- **agent に選んだテーマの文脈を与える。** 選択したテーマを与えて、あるテーマの専門家に育てられます。
- **モデルを訓練する。** 構造化された Meme で訓練すれば、生のテキストを読ませるより効率的で、はるかにトークンを節約できます。
- **自分の KB を育てる。** BusyBeaver を基礎知識の層として使い、自分自身のナレッジベース（たとえばあなたの wiki）を芽吹かせ、拡張できます。
- **統合し、発見する。** ここの知識は*クエリ可能*なので、知識ベースのデータ統合ができます——クエリを使ってセレンディピティを生み出し、探すべきと知らなかったつながりを浮かび上がらせます。

### 🌍 多言語ナレッジベース

BusyBeaverは**国際的**であるように構築されています。初期のコーパスは主に**英語と中国語**ですが、私たちは今後も世界の主要言語を追加し続けます。なぜなら、人類文明を形作った古典は、単一の言語だけで書かれたわけではないからです。

### 🔮 これから来るもの

BusyBeaverはコーパスです。その周りに、私たちは以下を構築しています。

- **すべての書籍のためのインタラクティブなwiki** — 単なる平坦なダンプではなく、生きたリンクされたwikiとして、各書籍の知識を閲覧・クエリできる。
- **あらゆるコンテンツ → wiki** — *任意の*コンテンツを、構造化されたエージェントネイティブなwikiに変換するサービス——**あなた自身の著作権のある素材**（社内ハンドブック、ライセンスされたテキスト、独自研究）を含む——これにより、あなた自身の専用KBを手に入れられる。
- **フェデレーテッド・セマンティックwiki** — 独立したwiki同士で知識エージェントが**自由に交換し、集約し、クエリ**できる共有プロトコル。一つのアドレス空間、多数の参加者——Purple Numberの夢の、フェデレーション版。

## 著作権と対象範囲

この公開ナレッジベースでは、**著作権のない（パブリックドメインの）書籍から得られた知識のみ**を公開しています。これにより、このダンプは共有しやすく、安心して土台にできる状態が保たれています。

あなた自身の**著作権のある素材**を、エージェントネイティブなKBに変換したいとお考えですか？それこそが**あらゆるコンテンツ → wiki**サービス（**これから来るもの**を参照）の目的です。早期の関心をお持ちの方は、**minnie.sew.da@gmail.com**までご連絡ください。

## 🚀 あなたの知識も、memify する

BusyBeaver は始まりにすぎません。私たちの計画は、**あなた自身の**知識——**個人のメモも企業のノウハウも**——を同じ種類の KB に変えるお手伝いをすることです：**配布可能で、組み合わせ可能で、agent が容易に消費できる**知識へ。

そしてそれは**あなたの管理下にあり続けます**。Meme ごとに異なる**可視性**を設定できます——あるものは非公開に、あるものはチームだけに共有し、あるものは世界へ公開する。あなたの知識を**特定の相手に向けて選択的に——あるいは有料で**、Meme 単位で配布できます。

古典は出発点にすぎません。より大きなビジョンは、信頼できる agent ネイティブな知識の、オープンで連合的な経済圏です——そこにあなたの知識の居場所があります。
